Constraints → complexity

Judges size n so exactly one complexity class survives the time limit — roughly 108 simple operations/second (C++-class constant factor; Python is often 5–20x slower, which is why LeetCode's limits are generous). Divide the time limit by that rate, find the complexity that fits.

Two constraints in one problem (grid n×m, or n elements / q queries) multiply: the real work is O(n·m) or O(n·q), not O(n).

Common gotchas

  • Sum-of-constraints across test cases. "1 ≤ T ≤ 1000, sum of n ≤ 2×10^5" means treat the sum as the real n, not the per-case bound.
  • Two-dimensional input. An n×m grid has n·m cells — a "linear scan" over it is O(n·m), not O(n).
  • Memory limit implies a complexity bound too. 256 MB ÷ 4 bytes/int ≈ 6×10^7 ints. A DP table exceeding that needs a rolling array or a different approach, independent of the time budget.
  • Amortized vs worst-case. Dynamic array push_back/append is amortized O(1) but worst-case O(n) on one call; Union-Find with path compression + union by rank is amortized O(α(n)) ≈ O(1).
  • Expected vs adversarial worst-case. Hash map lookups are expected O(1) but worst-case O(n) under collisions; quickselect is expected O(n) but worst-case O(n^2) without randomized pivoting. Matters on Codeforces, rarely on LeetCode.
  • Recursion depth is a hidden constraint. O(n) recursion at n ∼ 10^5 can blow the call stack before it blows the time limit — go iterative or check the stack size.
  • Constant factor still matters at the boundary. O(n log n) via a segment tree carries a much bigger constant than O(n log n) from a single sort(). Near the edge of a budget, prefer the simpler structure.

Generic pattern supersets

Narrow, named patterns nest inside broader families. When you don't recognize the exact pattern, recognizing the family narrows the search to 2–3 techniques instead of the full list above.

Window/pointer family

Linear scan with pointer(s) maintaining an invariant over a contiguous range.

Contains: two pointers (converging), sliding window (expand/shrink), fast/slow pointers (cycle detection).

Recognize from: array/string, contiguous, O(n) or O(n log n) expected, no need to revisit passed elements.

Search family

Exhaustive exploration of a state space, with or without pruning.

Contains: DFS ⊃ backtracking (DFS + undo on dead end). BFS is a sibling, not a subset — different traversal order, shortest-steps guarantee in unweighted graphs.

Recognize from: "all ways to...", "does a path exist...", tree/graph/grid input, n small enough for exponential blowup unless pruning/DP collapses it.

Graph traversal family

Superset of every graph algorithm.

Contains: DFS, BFS, Union-Find, topological sort, Dijkstra, Bellman-Ford, Floyd-Warshall, Kruskal/Prim.

Recognize from: explicit graph/tree/grid, or an implicit graph (states as nodes, transitions as edges — word ladder, puzzle BFS).

DP family — distinguished by state shape

Optimal substructure + overlapping subproblems. The sub-pattern is determined by the shape of the state, not the recurrence style.

  • Linear dp[i] — Kadane's, house robber, climbing stairs.
  • Two-sequence dp[i][j] — LCS, edit distance.
  • Interval dp[i][j] over range [i,j] — matrix chain multiplication, burst balloons, palindrome partitioning.
  • Subset dp[mask] — bitmask DP, TSP.
  • Tree dp[node][...] — tree DP via post-order DFS.
  • Digit dp[pos][tight][...] — count numbers in [L, R] with property X.

"Overlapping subproblems" is the easy part to spot — picking which of these six state shapes fits is 80% of a DP problem.

Range query family

Precompute or maintain an aggregate over ranges.

Contains: prefix sum (static, O(1) query) ⊂ difference array (static range updates, its write-side dual) ⊂ Fenwick/BIT (point update, range query, O(log n)) ⊂ segment tree (range update + range query, most general).

If there are no updates, prefix sums alone are enough — don't reach for a segment tree.

Stack/deque invariant family

Maintain a monotonic sequence to answer "nearest element with property X" in amortized O(1) per element.

Contains: monotonic stack (next/previous greater or smaller), monotonic deque (sliding window max/min — same invariant, but elements also fall off the front as the window slides).

How to simplify an unfamiliar problem

  1. Strip the flavor text. Identify: input shape (array / string / tree / graph / matrix / stream), constraints (n, value ranges, query count), question type (count / min / max / exists / construct / enumerate).
  2. Read the constraints before designing anything. They give the target complexity (see above) and eliminate most candidate families immediately — n ≤ 20 is very likely bitmask DP or backtracking, not a linear scan.
  3. Classify the input relationship. Is position/order load-bearing (array, string → window/pointer or sequence DP), or is it a relationship structure (graph, tree → traversal family)?
  4. Check for monotonicity. If "can we achieve X?" flips false→true exactly once as a parameter increases, that parameter is binary-searchable — binary search on the answer turns an optimization problem into an isolated decision check.
  5. Check for overlapping subproblems. If brute force recomputes the same (remaining_input, choices_so_far) state repeatedly, it's DP — next, pick the state shape.
  6. Look for a composition, not a single primitive. Most "creative" problems chain two ordinary patterns: sliding window + hashmap frequency count, binary search on the answer + greedy feasibility check, DFS + memoization (= tree DP), sort + two pointers.
  7. When stuck, measure the gap. Write the brute-force complexity and the complexity the constraints demand. The ratio tells you what to add: brute O(n^2) but n = 10^5 demands O(n log n) → you're missing a sort, a hashmap, or a binary search that removes a factor of n.

Further reading: AlgoMonster's 48-pattern curriculum covers the same idea at interview scale — DFS/BFS, two pointers, and sliding window alone reportedly cover a large share of FAANG-style interview problems. Use it as a cross-check, not a replacement for the supersets above.

Glossary

Big-O / Big-Theta / Big-Omega
O = upper bound, Ω = lower bound, Θ = both. Interview usage almost always means Θ even when people say "O."
Amortized complexity
Average cost per operation over a worst-case sequence. Dynamic array append: O(1) amortized, O(n) worst case on the resizing call.
Expected / average-case
Average over random inputs or randomized choices, distinct from worst-case. Hash map lookup: O(1) expected, O(n) worst case under adversarial collisions.
In-place vs auxiliary space
In-place uses O(1) extra memory beyond the input; auxiliary space is memory beyond the output.
Stable sort
Equal elements keep relative input order (merge sort is stable; typical quicksort/heapsort are not).
Memoization vs tabulation
Memoization: top-down recursion + cache. Tabulation: bottom-up, filling a table in dependency order. Same complexity, different control flow.
Greedy
Commit to the locally-best choice, never revisit. Correct only when local optimality provably implies global optimality; otherwise DP is the safe fallback.
Divide and conquer
Split into independent subproblems, solve recursively, combine. Unlike DP, subproblems don't overlap.
Meet in the middle
Split a 2^n search space into two 2^(n/2) halves, solve independently, combine — typically for n up to ~40.
Monotonic stack / deque
Kept strictly increasing/decreasing by popping violators before pushing; answers "nearest element satisfying X" in amortized O(1).
Prefix sum / difference array
Prefix sum: O(1) range-sum query on a static array. Difference array: its dual, O(1) range updates.
Fenwick tree (BIT)
Point update + prefix/range query in O(log n), small constant, simpler than a segment tree.
Segment tree
Range update + range query for any associative op in O(log n); more general than a Fenwick tree, bigger constant.
Union-Find (DSU)
Disjoint-set partition with near-O(1) union/find (amortized O(α(n))) using path compression + union by rank.
Topological sort
Linear ordering of a DAG's nodes so every edge points forward; undefined (and detectable) on a cycle.
Backtracking
DFS over a decision tree that undoes a choice on a dead end, optionally pruning to avoid full exponential exploration.
Bitmask
Represent a subset of up to ~20–24 items as an integer, one bit per item — the state representation for bitmask DP.
Binary search on the answer
Binary search over possible answers using a feasibility check per candidate; valid when feasibility is monotonic in the answer.
Heap / priority queue
O(log n) insert/extract-min-max, O(1) peek. Standard structure for top-k, k-way merge, Dijkstra's frontier.
Trie (prefix tree)
Tree where each root-to-node path spells a prefix; O(length) prefix queries for autocomplete, word search, prefix DP.
Kadane's algorithm
O(n) DP for max-subarray-sum: at each index, extend the running subarray or restart, whichever is larger.
Floyd's cycle detection
Tortoise/hare pointers at different speeds meet iff there's a cycle; further arithmetic locates its start in O(n) time, O(1) space.
Dijkstra vs Bellman-Ford vs Floyd-Warshall
Dijkstra: single-source, non-negative weights, O((V+E) log V). Bellman-Ford: single-source, handles negative weights, O(V·E). Floyd-Warshall: all-pairs, O(V^3), only for small V.
NP-hard (informal)
No known polynomial algorithm. If a problem is a disguised TSP/subset-sum/knapsack-decision/graph-coloring, expect exponential-but-bounded n, a pseudo-polynomial DP, or approximation.
